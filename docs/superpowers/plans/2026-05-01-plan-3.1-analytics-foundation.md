# Plan 3.1 — Analytics Foundation

Owner: Reese
Status: Implementation plan, ready for review
Predecessor: Plan 2 (Ingestion) — complete; 53 weeks imported as of 2026-04-30
Successor: Plan 3.2 (Read-only keyword explorer)

## Goal

Build the analytics-side data foundation that the explorer (Plan 3.2), keyword detail page (Plan 3.3), and watchlists (Plan 3.4) all depend on:

1. Two-tier fake-volume severity computed for every observed weekly metric row
2. A configurable rule table so admins can edit thresholds later without code changes
3. A denormalized `keyword_current_summary` table — one row per keyword, refreshed after every successful import, that powers fast explorer queries
4. The summary refresh job hooked into the import pipeline

No user-visible UI ships in 3.1. Output is purely data + jobs that future sub-phases consume.

## 2. Schema changes

### 2.1 New enum: `fake_volume_severity`

```sql
CREATE TYPE fake_volume_severity AS ENUM ('none', 'warning', 'critical');
```

`'none'` is used when the rule evaluated cleanly and neither tier matched. NULL (column nullable) is used when the rule could not be evaluated due to missing input (the existing `fake_volume_eval_status` enum captures the reason).

### 2.2 Add `fake_volume_severity` column to `keyword_weekly_metrics`

```sql
ALTER TABLE keyword_weekly_metrics
  ADD COLUMN fake_volume_severity fake_volume_severity;
```

Replaces the existing `fake_volume_flag boolean` column conceptually. The boolean column stays in the schema for now (NULL everywhere — never populated during backfill), and we'll drop it in Plan 3.5 (polish) once nothing reads it. Two-step migration is safer than rename-and-replace.

### 2.4 New `keyword_current_summary` table

One row per **currently-active** `search_term_id` — defined as terms whose `last_seen_week ≥ current_week_end_date - 28 days` (i.e., seen in the most recent 4 weeks of imported data). Refreshed after every successful import.

Why filter at 4 weeks (decided 2026-05-01 review):
- ~9.3M total unique terms accumulated across 53 weeks
- ~4M of those appear in any 4-week window — the rest haven't been observed for 5+ weeks
- Refresh INSERT time drops from ~10–20 min → ~5 min
- Explorer queries get ~2× faster (smaller table, smaller indexes)
- Long-dormant terms remain queryable via substring search (`search_terms` is unchanged) and via the keyword detail page (queries `kwm` directly). Only the default explorer browse view hides them.
- A term that reappears after dormancy gets re-included on the next refresh automatically.

```sql
CREATE TABLE keyword_current_summary (
  search_term_id uuid PRIMARY KEY REFERENCES search_terms(id),
  current_week_end_date date NOT NULL,
  current_rank integer NOT NULL,

  -- Historical comparison ranks (NULL = unranked or no data that far back)
  prior_week_rank integer,
  rank_4w_ago integer,
  rank_13w_ago integer,
  rank_26w_ago integer,
  rank_52w_ago integer,

  -- Improvement = prior_rank - current_rank (positive = improvement)
  improvement_1w integer,
  improvement_4w integer,
  improvement_13w integer,
  improvement_26w integer,
  improvement_52w integer,

  consecutive_improvement_streak integer NOT NULL DEFAULT 0,
  ever_top_50k boolean NOT NULL DEFAULT false,
  has_unranked_week boolean NOT NULL DEFAULT false,
  unranked_week_count integer NOT NULL DEFAULT 0,
  unranked_after_top_50k boolean NOT NULL DEFAULT false,

  last_seen_week date NOT NULL,
  weeks_since_seen integer NOT NULL DEFAULT 0,

  fake_volume_severity_current fake_volume_severity,

  -- "Currently looks like" snapshot of the latest observed week
  top_clicked_category_1_current varchar(255),
  top_clicked_product_1_asin_current varchar(20),
  top_clicked_product_1_title_current text,
  keyword_in_title_1_current boolean,
  keyword_in_title_2_current boolean,
  keyword_in_title_3_current boolean,
  keyword_title_match_count_current smallint,

  -- top_category_changed_recently / top_asin_changed_recently DEFERRED
  -- (decided 2026-05-01: low expected signal in low-volume keywords;
  -- add later via ALTER TABLE if user demand emerges)

  updated_at timestamptz NOT NULL DEFAULT now()
);
```

## 3. fake-volume severity backfill (one-time)

The existing 140M `keyword_weekly_metrics` rows have `fake_volume_severity = NULL` (the new column starts NULL by default). We need to compute it for every existing row.

### 3.1 Strategy: chunk by `week_end_date`

A single `UPDATE keyword_weekly_metrics SET ...` covering 140M rows would lock the table, generate massive WAL, and likely time out via Neon's HTTP driver. Instead we iterate over the 53 known weeks (one slice per week). Each weekly slice is ~2.7M rows.

The existing PK on `(week_end_date, search_term_id)` means filtering by `week_end_date = X` is a fast index range scan.

### 3.2 SQL per slice

```sql
UPDATE keyword_weekly_metrics
SET fake_volume_severity = CASE
  -- Eval impossible: keep NULL (existing fake_volume_eval_status captures why)
  WHEN top_clicked_product_1_click_share IS NULL
    OR top_clicked_product_1_conversion_share IS NULL
    THEN NULL

  -- Critical: stricter thresholds
  WHEN (top_clicked_product_1_click_share > 20
        AND top_clicked_product_1_conversion_share < 0.5)
    OR (top_clicked_product_1_click_share > 30
        AND top_clicked_product_1_conversion_share < 1.0)
    THEN 'critical'::fake_volume_severity

  -- Warning: looser thresholds
  WHEN (top_clicked_product_1_click_share > 5
        AND top_clicked_product_1_conversion_share < 0.5)
    OR (top_clicked_product_1_click_share > 10
        AND top_clicked_product_1_conversion_share < 1.0)
    THEN 'warning'::fake_volume_severity

  ELSE 'none'::fake_volume_severity
END
WHERE week_end_date = $1::date
  AND fake_volume_severity IS NULL;  -- idempotent — skip already-done rows
```

### 3.3 Backfill script

`scripts/backfillFakeVolumeSeverity.ts` — iterates the list of weeks from `reporting_weeks`, runs the slice UPDATE for each, prints progress. Run once after migration applies.

Estimated total time: 53 weeks × ~30s–60s per slice = 25–55 minutes.

### 3.4 Set `fake_volume_eval_status` consistently

For rows where `fake_volume_severity` is NULL because of missing input, also set `fake_volume_eval_status` to `unknown_missing_click` or `unknown_missing_conversion` (the existing enum). Currently this column is also NULL for everything — needs the same kind of slice-by-slice UPDATE.

Combine into one UPDATE per slice for efficiency.

## 4. fake-volume severity in the ongoing import pipeline

After 3.3 backfills the existing data, every NEW import needs to compute `fake_volume_severity` during the kwm INSERT. Two options:

### Option A (recommended): compute in the kwm INSERT directly

Modify the existing `kwm_insert` SQL in `processFileImport`:

```ts
INSERT INTO keyword_weekly_metrics (
  ...,
  fake_volume_severity,
  fake_volume_eval_status
)
SELECT
  ...,
  CASE WHEN s.top_clicked_product_1_click_share IS NULL
            OR s.top_clicked_product_1_conversion_share IS NULL
       THEN NULL
       WHEN ... THEN 'critical'::fake_volume_severity
       WHEN ... THEN 'warning'::fake_volume_severity
       ELSE 'none'::fake_volume_severity
  END AS fake_volume_severity,
  CASE WHEN s.top_clicked_product_1_click_share IS NULL THEN 'unknown_missing_click'
       WHEN s.top_clicked_product_1_conversion_share IS NULL THEN 'unknown_missing_conversion'
       ELSE 'evaluated'
  END::fake_volume_eval_status AS fake_volume_eval_status,
FROM staging_weekly_metrics s
JOIN search_terms st ON ...
WHERE s.uploaded_file_id = ${file.id}
ON CONFLICT (week_end_date, search_term_id) DO NOTHING
```

Pros: single INSERT, fewer phases, no extra UPDATE.
Cons: SQL is slightly more verbose; if the rule changes, this code needs to change.

### Option B: separate post-INSERT UPDATE phase

After `kwm_insert`, add a `fake_volume_eval` phase that UPDATEs the freshly inserted rows.

Pros: clean separation; the rule lookup happens in one place; easier to swap in a JSON-driven rule evaluator later.
Cons: extra DB round-trip; rewrites the same rows we just inserted (more WAL, dead tuples).

**Recommendation: Option A.** When admin-editable rules ship in Plan 3.5, we re-derive the SQL CASE from `fake_volume_rules.is_active` row's JSON and rebuild the SQL string at import time.

## 5. Summary refresh job

This is the meat of 3.1. Computes `keyword_current_summary` from kwm + reporting_weeks.

### 5.1 Approach: build via temp tables, then atomic swap

A direct INSERT...SELECT into `keyword_current_summary` covering all eligible terms with multiple JOINs to kwm and complex window functions, all in one statement, is slow and hard to debug.

Better: stage each piece into temp tables, then build final rows from staged pieces, then atomically replace `keyword_current_summary` contents.

The 4-week active-term filter is applied at the FIRST step (`latest_per_term`) so all subsequent staging tables are scoped to the ~4M currently-active terms — much faster downstream.

### 5.2 Steps

```sql
-- The "current week" for the whole refresh is the max week in reporting_weeks.
-- All "weeks_since_seen" / "rank_Nw_ago" calculations anchor on this date.
-- Materialized as a single date constant to avoid recomputing.

-- Step 1: latest_per_term — current week's row per search_term,
--         FILTERED to terms seen in the last 4 weeks (28 days).
CREATE TEMP TABLE latest_per_term AS
SELECT DISTINCT ON (k.search_term_id) k.*
FROM keyword_weekly_metrics k,
     (SELECT MAX(week_end_date) AS current_week FROM reporting_weeks WHERE is_complete) ref
WHERE k.week_end_date >= ref.current_week - INTERVAL '28 days'
ORDER BY k.search_term_id, k.week_end_date DESC;
-- Expected: ~4M rows (vs ~9.3M without the filter)

-- Step 2: rank_at_offset(N) — rank N weeks before each term's current_week
-- "N weeks ago" = the row whose week_end_date <= current_week - N*7 days,
-- taking the most recent qualifying row.
-- Generate one temp table per offset (1, 4, 13, 26, 52).
CREATE TEMP TABLE rank_at_1w AS
SELECT k.search_term_id, k.actual_rank
FROM keyword_weekly_metrics k
JOIN latest_per_term l USING (search_term_id)
WHERE k.week_end_date <= l.week_end_date - INTERVAL '7 days'
  AND k.week_end_date >= l.week_end_date - INTERVAL '13 days'  -- bounded for index range
ORDER BY search_term_id, week_end_date DESC
LIMIT 1;
-- ... same pattern for 4w, 13w, 26w, 52w

-- Step 3: per-term aggregates
CREATE TEMP TABLE aggs_per_term AS
SELECT
  search_term_id,
  COUNT(*) AS weeks_present,
  MIN(actual_rank) AS best_rank_ever,
  BOOL_OR(actual_rank <= 50000) AS ever_top_50k
FROM keyword_weekly_metrics
GROUP BY search_term_id;

-- Step 4: streak computation (window function)
CREATE TEMP TABLE streaks AS
WITH ranked AS (
  SELECT search_term_id, week_end_date, actual_rank,
         LAG(actual_rank) OVER (PARTITION BY search_term_id ORDER BY week_end_date) AS prev_rank
  FROM keyword_weekly_metrics
),
improving AS (
  SELECT search_term_id, week_end_date,
         CASE WHEN prev_rank IS NULL THEN 0
              WHEN prev_rank > actual_rank THEN 1
              ELSE 0
         END AS improved
  FROM ranked
),
groups AS (
  SELECT search_term_id, week_end_date, improved,
         SUM(CASE WHEN improved = 0 THEN 1 ELSE 0 END)
           OVER (PARTITION BY search_term_id ORDER BY week_end_date) AS streak_group
  FROM improving
),
streak_counts AS (
  SELECT search_term_id, streak_group,
         COUNT(*) AS streak_len,
         MAX(week_end_date) AS streak_end
  FROM groups
  WHERE improved = 1
  GROUP BY search_term_id, streak_group
)
SELECT DISTINCT ON (search_term_id) search_term_id, streak_len AS current_streak
FROM streak_counts
ORDER BY search_term_id, streak_end DESC;

-- Step 5: TRUNCATE + INSERT (atomic-ish via single transaction)
BEGIN;
TRUNCATE keyword_current_summary;

INSERT INTO keyword_current_summary (
  search_term_id, current_week_end_date, current_rank,
  prior_week_rank, rank_4w_ago, rank_13w_ago, rank_26w_ago, rank_52w_ago,
  improvement_1w, improvement_4w, improvement_13w, improvement_26w, improvement_52w,
  consecutive_improvement_streak, ever_top_50k,
  -- ... etc
)
SELECT
  l.search_term_id,
  l.week_end_date,
  l.actual_rank,
  r1w.actual_rank,
  r4w.actual_rank,
  r13w.actual_rank,
  r26w.actual_rank,
  r52w.actual_rank,
  COALESCE(r1w.actual_rank - l.actual_rank, NULL),
  COALESCE(r4w.actual_rank - l.actual_rank, NULL),
  COALESCE(r13w.actual_rank - l.actual_rank, NULL),
  COALESCE(r26w.actual_rank - l.actual_rank, NULL),
  COALESCE(r52w.actual_rank - l.actual_rank, NULL),
  COALESCE(s.current_streak, 0),
  a.ever_top_50k,
  -- ... etc
FROM latest_per_term l
JOIN aggs_per_term a USING (search_term_id)
LEFT JOIN streaks s USING (search_term_id)
LEFT JOIN rank_at_1w r1w USING (search_term_id)
LEFT JOIN rank_at_4w r4w USING (search_term_id)
LEFT JOIN rank_at_13w r13w USING (search_term_id)
LEFT JOIN rank_at_26w r26w USING (search_term_id)
LEFT JOIN rank_at_52w r52w USING (search_term_id);

COMMIT;
```

**Note**: Steps 4 (streak) and 6 (unranked counts, top_category_changed_recently, top_asin_changed_recently) are the most complex. We'll implement them iteratively — ship 3.1 with the simpler fields populated and the harder ones as `NOT NULL DEFAULT 0/false`, then enrich in Plan 3.5.

### 5.4 Estimated cost per run (with 4-week active filter)

Each component:
- `latest_per_term` — DISTINCT ON across kwm rows in a 28-day window (~10M rows of the 140M total). Index range scan on PK `(week_end_date, search_term_id)`. Output: ~4M rows.
- `rank_at_Nw` × 5 — each joins `latest_per_term` (4M) to kwm at the offset. Uses `kwm_term_week_idx (search_term_id, week_end_date)`. ~30s each.
- `aggs_per_term` — full kwm scan grouped by search_term_id BUT scoped to the 4M active terms via JOIN. ~30s on hot cache.
- `streaks` — window function ordered by `(search_term_id, week_end_date)`, scoped to active terms. ~1 min.
- TRUNCATE + INSERT 4M rows — TRUNCATE is instant; INSERT maintains 9 indexes. ~2–3 min.

**Total estimated**: 5–8 minutes per refresh (down from 10–20 min with no filter).

This is significant. Three optimizations to consider for Plan 3.5:

- **Incremental refresh**: instead of full TRUNCATE+INSERT, only refresh the rows whose `current_week_end_date < latest_imported_week`. Most terms didn't change between imports.
- **Materialized view + REFRESH MATERIALIZED VIEW CONCURRENTLY**: lets queries continue reading the old version while the new version builds. Eliminates the brief gap during refresh.
- **Drop indexes during refresh, rebuild after**: the same trick we used for backfill.

For 3.1, we do the simple TRUNCATE+INSERT. Optimize later.

## 6. Hooking into the import pipeline

After Plan 3.1 ships, `processFileImport` gains a final phase:

```ts
async function processFileImport(input: ImportFileInput): Promise<ImportFileOutput> {
  // ... lock, COPY, search_terms_upsert, kwm_insert (with new fake_volume_severity column)

  // Existing final phases:
  // - reporting_weeks_upsert
  // - staging_cleanup
  // - mark_imported

  // NEW: summary refresh
  await timePhase(file.id, 'summary_refresh', () =>
    refreshKeywordCurrentSummary(),
  );

  // (mark_imported moves here, AFTER summary refresh)
}
```

Adds 10–20 minutes to each weekly import. Acceptable given imports already take 20–30 minutes.

If the summary refresh fails for any reason, the file is marked `import_failed` and the kwm/staging state is left consistent (kwm rows are in, but `keyword_current_summary` may show stale data). Recovery = manually retry the refresh job; data is safe.

## 8. Testing approach

For 3.1:

- **Unit-ish via spot SQL**: after each migration step, run a SELECT to verify column / table / row exists.
- **fake_volume_severity backfill**: query a random sample of 100 kwm rows where severity was just computed; manually verify the rule was applied correctly across all 4 cases (none / warning / critical / NULL).
- **summary refresh**: run once, verify `keyword_current_summary` has 9.3M rows, spot-check 10 rows against manual computation from kwm.
- **import pipeline**: re-import one fresh week of data, confirm `fake_volume_severity` is populated for the new rows and `keyword_current_summary` is refreshed.

No E2E test framework added in 3.1 — manual SQL spot checks are sufficient before user-facing UI exists.

## 9. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Backfill UPDATE per slice times out via Neon's HTTP driver | Medium — slices are 2.7M rows each | Use `pg.Pool` (TCP keepalive) for backfill script, not `@neondatabase/serverless` HTTP. Consistent with our `processFileImport` approach. |
| `latest_per_term` query is slow without supporting index | Low | `kwm_term_week_idx` (search_term_id, week_end_date) covers it |
| Summary refresh adds 10–20 min to each weekly import | High (it does!) | User accepts per response above |
| `keyword_current_summary` indexes slow down the refresh INSERT | Medium | Drop indexes pre-INSERT, recreate after — Plan 3.5 optimization |
| Streak/unranked computation is buggy on first try | Medium | Ship 3.1 with simpler placeholders (streak=0, unranked_week_count=0); enrich in 3.5 |
| Summary refresh failure leaves stale data | Low impact | mark_imported runs AFTER refresh — file shows imported only when summary matches kwm |

## 10. Sequencing within Plan 3.1

Suggest implementing in this order, each step verifiable independently:

1. **Migration + seed**: tables, enum, indexes, default rule row. Verify schema. ~30 min.
2. **`backfillFakeVolumeSeverity.ts`**: run against existing 140M rows. ~25–55 min.
3. **`refreshSummary.ts` job**: write the SQL, run it manually once via `refreshSummaryOnce.ts`. Verify 9.3M rows produced. ~10–20 min compute, plus an hour or two of writing and debugging the SQL.
4. **Import pipeline integration**: add `fake_volume_severity` to `kwm_insert`, add `summary_refresh` phase. Test by importing a fresh week (we can do a single test file).
5. **Drizzle schema + index file**: bring all the new tables into the Drizzle types so future code can reference them.

I'll commit each step as its own commit so we can revert if anything goes sideways.
