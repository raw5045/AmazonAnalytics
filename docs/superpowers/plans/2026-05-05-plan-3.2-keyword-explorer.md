# Plan 3.2 — Read-only Keyword Explorer

Owner: Reese
Status: Implementation plan, ready for review
Predecessor: Plan 3.1 (Analytics Foundation) — complete; full-history fake_volume backfill done; keyword_current_summary populated with 3.84M active terms
Successor: Plan 3.3 (Keyword Detail Page)

## Goal

Ship the **first user-visible analytics page**: the keyword explorer. Read-only at this stage — saved views and watchlists land in Plan 3.4, alerts in 3.5. This is the page where users (admins + standard users) actually browse the imported keyword data.

Reads exclusively from `keyword_current_summary` + `search_terms` (no kwm scans on the hot path). Search-term substring matching uses the existing GIN trigram index on `search_terms.search_term_normalized`.

## Out of scope (explicitly)

- Saved views, watchlists, watchlist alerts → Plan 3.4
- Keyword detail page (drill-in from explorer row) → Plan 3.3
- Streak / `ever_top_50k` / unranked flag filters — those columns are defaulted to 0/false in `keyword_current_summary`. The explorer's UI doesn't expose filters for them in 3.2; we'll wire them up after they're properly populated in Plan 3.5.
- Mobile-optimized layout. Desktop-first; mobile is "works but not polished" in 3.2.
- Bulk export / CSV download.

## 1. Filters (V1)

Match the spec where it makes sense, defer the parts that depend on deferred fields.

### 1.1 Window selector

Single-select: `1 week / 4 weeks / 13 weeks / 26 weeks / 52 weeks`. Drives:
- Which `prior_*_rank` and `improvement_*` columns to read
- Which `improvement_*` index gets used for sorting

UI label: `Week / Month / 3 Months / 6 Months / Year`. URL param: `?window=1w|4w|13w|26w|52w`.

### 1.2 Search term contains (substring)

Free-text input. Maps to:
```sql
... WHERE st.search_term_normalized LIKE '%' || $pattern || '%'
```
Uses the GIN trigram index that already exists. Fast for patterns ≥ 3 chars.

For patterns < 3 chars we display "Type at least 3 characters to filter by text" and skip the WHERE clause.

URL param: `?q=`.

### 1.3 Current rank min/max

Two number inputs. Maps to `current_rank BETWEEN $min AND $max`. Uses `kcs_rank_idx` on `(current_week_end_date, current_rank)`.

Defaults: min=1, max=unlimited. URL params: `?rank_min=1&rank_max=`.

### 1.4 Threshold jump (4 presets)

Single-select dropdown with 4 predefined patterns + "none" default. Works in combination with the window selector (1.1) to find keywords that crossed a major rank threshold over the chosen window.

| Preset | SQL |
|---|---|
| (none — default) | no filter |
| "Outside top 500k → inside top 100k" | `rank_${window}_ago > 500000 AND current_rank < 100000` |
| "Outside top 100k → inside top 50k" | `rank_${window}_ago > 100000 AND current_rank < 50000` |
| "Outside top 100k → inside top 10k" | `rank_${window}_ago > 100000 AND current_rank < 10000` |
| "Outside top 50k → inside top 10k" | `rank_${window}_ago > 50000 AND current_rank < 10000` |

URL param: `?jump=500k_to_100k|100k_to_50k|100k_to_10k|50k_to_10k`.

Future Plan 3.5 work: free-form custom thresholds. The 4 presets cover the common cases users typically want (per-spec).

### 1.5 Top clicked category #1

Single-select dropdown. Categories populated from `SELECT DISTINCT top_clicked_category_1_current FROM keyword_current_summary` (cached for 1 hour). Maps to `top_clicked_category_1_current = $category`.

Uses `kcs_category_idx`. URL param: `?category=`.

### 1.6 Fake volume severity filter

Three-state checkbox group:
- ☑ `none` — show clean keywords (default on)
- ☑ `warning` — show orange-flagged (default on)
- ☐ `critical` — show red-flagged (default off — hides obvious fake-volume noise)

Maps to `fake_volume_severity_current = ANY($severities)` (or includes NULL when 'none' is checked, since NULL means "couldn't evaluate").

URL param: `?severity=none,warning` (comma-separated).

### 1.7 Title-gap filter

Two parts:

**Part A — which title slots to include:** checkbox group for #1, #2, #3. Default all three on. URL param: `?titles=1,2,3`.

**Part B — match mode:**
- "Show all" (no title filter)
- "Missing from any selected"
- "Missing from all selected"

Maps to:
```sql
-- Missing from any selected (title 1+2+3 all on, default)
WHERE NOT (keyword_in_title_1_current
        AND keyword_in_title_2_current
        AND keyword_in_title_3_current)

-- Missing from all selected
WHERE NOT keyword_in_title_1_current
  AND NOT keyword_in_title_2_current
  AND NOT keyword_in_title_3_current
```

URL param: `?title_match=any|all`.

## 2. Sort options

Top of results table, single-select dropdown. Default is **best current rank ASC** (lowest SFR = most-searched, most relevant first).

| Sort | SQL |
|---|---|
| **Best current rank (default)** | `ORDER BY current_rank ASC` |
| Worst current rank | `ORDER BY current_rank DESC` |
| Biggest improvement (window-relative) | `ORDER BY improvement_${window} DESC NULLS LAST` |
| Biggest decline (window-relative) | `ORDER BY improvement_${window} ASC NULLS LAST` |
| Title gap (most gaps first) | `ORDER BY keyword_title_match_count_current ASC` |

Uses the corresponding indexes from Plan 3.1.

URL param: `?sort=rank|rank_desc|imp|decline|title_gap`.

(Alphabetical sort by category not included — user confirmed not needed for V1.)

## 4. Pagination

- Default page size: **100 rows** (per user decision: "show all the filter options with nothing applied along with the top 100 keywords")
- Show "Showing 1–100 of 12,345 — page 2 of 124" + Prev/Next/Jump-to-page
- URL params: `?page=2&per_page=100`
- Use `LIMIT $perPage OFFSET ($page - 1) * $perPage` — fine for paged browsing up to ~10k pages on the indexed sort columns. Spec doesn't require keyset pagination yet.
- Initial empty state (no URL params): all filters in their default state, results table shows top 100 keywords sorted by `current_rank ASC`.

For very deep pagination (page 1000+) we'd want keyset; defer to 3.5 if it becomes a problem.

## 5. Auth gating change

Currently every `/admin/*` route requires `admin` role. Plan 3.2 introduces the first analytics route accessible to `standard_user`:

- Route: `/explorer` (NOT under `/admin`)
- Both `admin` and `standard_user` can access
- `requireAuthenticatedUser()` helper (vs. the existing `requireAdmin()`) — new helper to write
- Middleware update: `/explorer` and any sub-routes go through the standard auth check, not the admin check

Same Clerk integration; just a different gate.

## 6. URL state contract

The explorer's full state must be encoded in URL params so:
- Back/forward buttons work
- Filter state survives a refresh
- Users can share a link to a filtered view (foundation for Plan 3.4 saved views)

Server component reads from `searchParams`, builds the SQL, renders. Client filter components mutate `searchParams` via `router.replace()`. No client-side state for filter values — URL is single source of truth.

## 8. Indexes — review against Plan 3.1's existing set

From Plan 3.1:

| Index | Used by which filter combination |
|---|---|
| `kcs_rank_idx (current_week_end_date, current_rank)` | rank min/max + sort by rank |
| `kcs_title_match_idx (current_week_end_date, keyword_title_match_count_current)` | sort by title_gap |
| `kcs_imp1_idx ... kcs_imp52_idx (improvement_Nw)` | sort by improvement |
| `kcs_category_idx (current_week_end_date, top_clicked_category_1_current)` | category filter |
| `kcs_severity_idx (current_week_end_date, fake_volume_severity_current)` | severity filter |

**Four new composite indexes for threshold-jump filters (create as part of Plan 3.2):**

Threshold-jump is a featured filter in V1, so we want the indexes in place before users start hitting the page rather than discovering slowness post-launch.

```sql
-- Composite for "rank_Nw_ago > X AND current_rank < Y" queries.
-- Index ordering matches the filter pattern: rank_Nw_ago first (the OUTSIDE
-- condition has higher selectivity at large thresholds), current_rank second.
CREATE INDEX kcs_jump_4w_idx
  ON keyword_current_summary (rank_4w_ago, current_rank);
CREATE INDEX kcs_jump_13w_idx
  ON keyword_current_summary (rank_13w_ago, current_rank);
CREATE INDEX kcs_jump_26w_idx
  ON keyword_current_summary (rank_26w_ago, current_rank);
CREATE INDEX kcs_jump_52w_idx
  ON keyword_current_summary (rank_52w_ago, current_rank);
```

(`rank_1w_ago` skipped — 1-week threshold jumps are rare and not in the 4 presets.)

Migration goes alongside the click_share/conversion_share schema additions in the same Plan 3.2 migration. Total of 6 schema deltas: 2 new columns + 4 new indexes.

## 10. Open questions

1. **Default filters when arriving with no URL params**: show everything? Or apply default exclusions (e.g., severity = none + warning)? **Recommend: show everything by default**, let user discover filters.

2. **Empty state**: when filters return 0 results, what do we show? Suggest a "Try removing a filter" hint with a "Reset filters" link. **Recommend: yes, build it.**

3. **Counts of filter values**: for the category dropdown, do we show counts ("Electronics (1,234)")? Adds a slow `GROUP BY` query per page load. **Recommend: skip counts in 3.2**; just show category names.

4. **What's "Top product #1" rendered as**: title text linking to Amazon? Or just text + ASIN tooltip? **Recommend: text + ASIN tooltip; no Amazon link in 3.2** (avoid affiliate-link complications until we decide policy).

5. **Search-term display**: raw or normalized? Spec says raw (preserves user's actual query). **Recommend: raw**, with normalized shown as a secondary line if they differ visibly.
